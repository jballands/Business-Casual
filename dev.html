The Business Casual Architecture

In general, the architecture of Business Casual can be broken down into three parts:

  1. The development part
  2. The building part
  3. The production part

You can think of the building part as the ice cream in an ice cream sandwich, binding the two chocolate cake pieces together. Both the development part and the production part define the same rules and instructions; the way those pieces appear, however, are very different.

The Development Part

This part of the Business Casual workflow is where you'll actually write code. The development part is written in two programming languages: LESS and Javascript.

Pro tip
LESS is a CSS pre-processor that allows you to use logic, control paths, and variables in your stylesheets. Once you have written LESS code, you need to compile that code into CSS in order for it to work in the browser. Click here to learn more.

LESS code exists in the less/ directory and is broken down into 11 files. Out of these files, 4 of them are important in order to understand the structure of Business Casual:

File Description
basics.less Defines basic HTML features that are styled in Business Casual, like h1 and code tags.
business-casual.less The "god file" that imports all other LESS files.
customize.less Defines all customizable attributes for easy modification. This file is intended to be modified.
scaffolding.less Defines the default values for the values in customize.less. This file is not intended to be modified.

If you wish to add another LESS file to extend Business Casual, by all means do so; just remember to add an import statement in business-casual.less.

Heads up!
The order of the import statements in business-casual.less matter! When you add your own import statement, you should always add it to the end of business-casual.less for safety.

Javascript code exists in the js/ directory under the file business-casual.js and is minimized directly by Grunt. Since this file has a very particular stucture to it, we will discuss this file in detail in the Javascript section.

The Building Part

Once you have finished writting Javascript and LESS code, it's time to "build" Business Casual by running it through a task runner known as Grunt. "Building" refers to the process of verifying, consolidating, and shrinking your code to something smaller and more manageable.

Grunt is a piece of software that automates the building process for you. Grunt is written in Javascript (traditionally a language for web browsers) and therefore must run on a platform called Node. The process for installation is described on the main Business Casual website.

Grunt uses two plugins to build your code: lessc and Uglify. lessc is a LESS compiler that takes all your LESS files and consolidates them into a single CSS file. Uglify is a Javascript compiler that minimizes your Javascript to reduce its file size and loading times. These plugins are installed using the npm install command on your command line.

Pro tip
Uglify gets its name because minimized Javascript is completely incomprehensible, i.e. ugly.

The Production Part

This is the part of the Business Casual workflow that you will use to style your website. There are only two files that you have to include in your HTML code to style it with Business Casual: business-casual.css and business-casual.min.js.

Both business-casual.css and business-casual.min.js are much smaller than their developmental counterparts. Grunt will report exactly how much your code was reduced when you run it. For example, LESS starting at 22.45kB will shrink to 18.21kB, and Javascript starting at 21.44kB will shrink to 7.56kB. This is beneficial because it reduces the amount of dependencies in your website and it allows your website to load much faster.

Business Casual Javascript

If you ever got to look at older versions of business-casual.js, you knew that the code was complicated and hard to extend. Starting with BC v1.2, business-casual.js has been completely refactored to alleviate many of problems encountered when working with BC JS. Some of the design goals for the refactoring of BC JS are described below:

  • Create a one-to-one relationship between DOM elements and JS objects
  • Abstract out the complicated parts of Javascript programming to make BC easier to extend
  • Force JS objects to interact with browser properties through a well-defined JS framework
  • Prevent spaghetti code from occuring

The following sections briefly discuss the different parts of BC JS and how you can interact with them to add your own dynamic features to Business Casual.

Overview

In general, BC JS is made up of four important components:

  • The BusinessCasual object
  • The Service objects
  • The Broadcaster object
  • BC DOM objects

Each of these four components work together in order to create dynamic features in Business Casual. Let's discuss the role of each component in Business Casual and how they work:

The BusinessCasual Object

The BusinessCasual object is a singleton object that acts the controller for all of Business Casual. Each of the BC DOM objects are invoked via the BusinessCasual object, and the entire bootstrapping and execution process is choreographed by this object.

Properties

version The version that this particular business-casual.js file conforms to.
availables A dictionary of possible queries and their corresponding BC DOM objects. It is because of this dictionary that the BusinessCasual object knows how to form a one-to-one relationship between BC DOM objects and real DOM objects.
elements An array of all the BC DOM objects that are currently in existance.

Functions

bootstrap Browses the availables dictionary and makes a query on the document. If the document returns any matches, this function will instatiate the corresponding BC DOM object and append it into the elements array.
run Loops through each element in the elements array and calls execute on it.

What's Important?

  1. The BusinessCasual object acts as the controller for BC JS.
  2. The bootstrap function is the same function that you call in script tags in your HTML code.
  3. This object uses its availables dictionary to make one-to-one relationships between BC DOM objects and real DOM objects.

The BC DOM Objects

BC DOM objects define the behavior of an instance of a particular dynamic feature in Business Casual. For example, there exists a sticky BC DOM object to define the behavior of the sticky feature. There can exist infinitely many BC DOM objects, and exactly one instance of a BC DOM object will be mapped to exactly one instance of its corresponding real DOM object. So, if there are 5 sticky elements, there will be 5 sticky BC DOM objects, where each BC DOM object is completely self-contained.

Functions

Constructor Each BC DOM object needs to implement its own constructor. It is here that the BC DOM object will describe what it needs to do when the BusinessCasual object asks it to bootstrap to a particular element in the DOM. In other words, any initialization that needs to be done during the bootstrapping process needs to be performed in this function.

The constructor must take exactly one argument called elem, which is the element that the BusinessCasual object is asking this object to bootstrap itself to.
execute When the webpage finishes bootstrapping, the BusinessCasual object will then begin calling this function on each BC DOM object in existance. It is here that the BC DOM object will describe what it needs to do when the webpage is running. This function takes no arguments.

What's Important?

  1. For each instance of a dynamic feature, there will exist exactly one instance of its corresponding BC DOM object.
  2. The BusinessCasual object will instantiate as many instances of a BC DOM object as it needs and automatically call the appropriate functions on it.
  3. Each of the described functions are only executed once by the BusinessCasual object. It is the responsibility of the BC DOM object to retrieve the approriate service for indefinite operation.
  4. The constructor must take exactly one argument called elem, the element that the BusinessCasual object is asking this object to bootstrap itself to. The execute function must take no arguments.

The Service Objects

Javascript allows you to assign functions to browser properties, like onscroll for example, so that you can define your own custom functionalit to them. Unfortunately, you can only assign a single function to a browser property, which means performing multiple tasks for a browser property can get a little crazy.

The Service objects in BC JS attempt to alleviate this problem by providing a safe route for BC DOM objects to take when they want to utilize a particular browser property.

Properties

subscribers Functions that were provided by instances of BC DOM objects that perform an action the BC DOM object wants based on this service.

Functions

subscribe Given a function func, adds it to a subscriber list to get invoked when the browser property is called. For example, in the ScrollingService, functions may subscribe to the service to get invoked when the webpage scrolls.

What's Important?

  1. Services are simply wrappers around browser properties that make it safe for multiple instances of BC DOM objects to touch a browser variable at the same time.
  2. All services are static prototypes. So, to subscribe to the ScrollingService, simply call ScrollingService.subscribe(func).
  3. Services take functions provided by instances of BC DOM objects and queue them for execution when the browser property is invoked.
  4. You cannot unsubscribe from a service once you have subscribed to it.

The Broadcaster Object

The Broadcaster object is a singleton object that allows instances of a BC DOM object to share variables amongst themselves. In other words, the Broadcaster provides a safe way to define global variables.

Properties

channels Key-value pairs that the Broadcaster object references.

Functions

hasChannel Given a channel c, checks for the existance of the channel in the channels property.
getChannel Given a key name, retrieves the value corresponding to the given name.
setChannel Given a key name and a value value, creates a channel in the channels property.

What's Important?

  1. The Broadcaster object is a singleton that allows multiple BC DOM objects to safely access global varaibles.
  2. The Broadcaster is a static prototype. So, to ask the Broadcaster if there is a channel named foo, simply call Broadcaster.hasChannel("foo").
  3. Never define your own variables at global scope in business-casual.js; you should always use the Broadcaster for this purpose.
  4. You should always call hasChannel before getChannel to avoid undefined variables.
